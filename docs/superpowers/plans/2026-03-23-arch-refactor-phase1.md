# Architecture Refactor — Phase 1: Database & ORM Layer

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Unify all router database access under a single `get_db()` Session dependency and replace the manual `_migrate()` migration system with Alembic.

**Architecture:** Remove `get_raw_db()` from `deps.py`; migrate the four routers that use raw `sqlite3.Connection` to use `Session = Depends(get_db)` instead. Routers that own their SQL use `session.execute(text(...))`. Routers that delegate to the legacy `engine.py` analytics module extract the underlying DBAPI connection via `backend.database.engine.raw_connection()` — this bridge is isolated and documented as technical debt. The `_migrate()` DDL + seed logic is split: DDL becomes the initial Alembic migration (stamped `head`); data seeds move to `backend/seeds.py` called on startup.

**Tech Stack:** SQLModel 0.0.21, SQLAlchemy 2.x, Alembic (new), Python 3.x, pytest.

## Key Design Decisions

### Why `engine.raw_connection()` for dashboard.py and equity.py GET?

`engine.py` is a 623-line analytics module that uses `sqlite3.Row` dict-style access throughout. Refactoring it to accept `Session` is a large separate task. The bridge pattern used here:

```python
from backend.database import engine as _sa_engine
import sqlite3

raw = _sa_engine.raw_connection()   # borrows from the SQLAlchemy connection pool
raw.row_factory = sqlite3.Row
try:
    result = build_summary(raw)
finally:
    raw.close()                      # returns the connection to the pool
```

This is correct, pool-safe, and self-documenting. It does not conflict with the `Session = Depends(get_db)` on the same handler — the session is unused on these read-only endpoints but satisfies the architectural contract. A TODO comment in each handler flags it for future cleanup.

### Why stamp head instead of running the initial migration?

The production `finance.db` already has all tables created by `create_db_tables()` + all the `ALTER TABLE` changes from `_migrate()` (v2–v9). Running the autogenerated migration would attempt to `CREATE TABLE` on existing tables and fail. `alembic stamp head` records the current revision in the `alembic_version` table without executing any DDL — exactly right for adopting Alembic into a live database.

### Seed extraction strategy

`_migrate()` mixes DDL (ALTER TABLE) and DML (INSERT). After Alembic takes over DDL:
- ALTER TABLE changes are represented in the initial migration (current schema = final state)
- INSERT seeds (v4 routing_targets, v4 categories backfill, v6 TaxProfile, v7 UserProfile, v8 Ledger/LedgerAccess) move to `backend/seeds.py::run_seeds(conn)`
- Seeds are idempotent (`INSERT OR IGNORE`) — safe to call on every startup
- `run_seeds()` accepts a raw sqlite3.Connection (same as current `_migrate`) so `ingest.py` and test teardowns work without touching Alembic

  from sqlalchemy import engine_from_config, pool
  from alembic import context
  import sys, os

  # Make sure 'backend' package is importable when alembic is run from project root
  sys.path.insert(0, os.path.dirname(os.path.dirname(__file__)))

  # Import all SQLModel table classes so their metadata is registered
  import backend.models  # noqa: F401 — registers all table=True classes
  from sqlmodel import SQLModel

  config = context.config
  if config.config_file_name is not None:
      fileConfig(config.config_file_name)

  target_metadata = SQLModel.metadata


  def run_migrations_offline() -> None:
      url = config.get_main_option("sqlalchemy.url")
      context.configure(
          url=url,
          target_metadata=target_metadata,
          literal_binds=True,
          dialect_opts={"paramstyle": "named"},
          render_as_batch=True,   # required for SQLite ALTER TABLE support
      )
      with context.begin_transaction():
          context.run_migrations()


  def run_migrations_online() -> None:
      connectable = engine_from_config(
          config.get_section(config.config_ini_section, {}),
          prefix="sqlalchemy.",
          poolclass=pool.NullPool,
      )
      with connectable.connect() as connection:
          context.configure(
              connection=connection,
              target_metadata=target_metadata,
              render_as_batch=True,   # required for SQLite ALTER TABLE support
          )
          with context.begin_transaction():
              context.run_migrations()


  if context.is_offline_mode():
      run_migrations_offline()
  else:
      run_migrations_online()
  ```

  def init_db(db_path: str | Path = DB_PATH) -> sqlite3.Connection:
      """
      Open the SQLite database with WAL mode and FK enforcement.
      Returns an open connection with row_factory set to sqlite3.Row.

      When called with an alternate path (e.g. ':memory:' in tests), bootstraps
      the schema via SQLModel.metadata.create_all() so tests remain self-contained.
      DDL for the production database is managed by Alembic.
      """
      path_str = str(db_path)
      conn = sqlite3.connect(path_str, check_same_thread=False)
      conn.row_factory = sqlite3.Row
      conn.execute("PRAGMA journal_mode=WAL;")
      conn.execute("PRAGMA foreign_keys=ON;")

      if path_str != str(DB_PATH):
          from sqlalchemy import create_engine as _sa_create_engine
          _tmp_engine = _sa_create_engine("sqlite://", creator=lambda: conn)
          SQLModel.metadata.create_all(_tmp_engine)

      return conn
  ```

## Gotchas & Risk Notes

| Risk | Mitigation |
|---|---|
| Alembic autogenerate may miss some columns (SQLModel quirks with Optional fields) | Review the generated migration carefully in Step 2.5 before stamping |
| `render_as_batch=True` is required for SQLite — without it, `alembic upgrade` on a migration with column changes will fail | Already included in `env.py` template above |
| `engine.raw_connection()` is a pool borrow — always call `.close()` in a finally block | Pattern shown in Steps 1.4 and 1.5 |
| SQLAlchemy RowMapping `.column_name` access vs `sqlite3.Row` `["column_name"]` — both work the same for simple selects | Verified in debt.py and settings.py patterns above |
| `alembic stamp head` requires the `alembic_version` table — Alembic creates it automatically | No action needed |
